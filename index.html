The Customization Stack

Three layers that build on each other. MCP connects to services. Skills teach processes. Plugins package everything for distribution.

Product Layer
Plugins
Complete, installable workflow packages. Bundle skills + connectors + slash commands + sub-agents. One-click install from a marketplace.
Code Chat Cowork
Instruction Layer
Skills
Portable instruction sets (SKILL.md files) that teach Claude specific processes. Same format works everywhere. Open standard at agentskills.io.
Code Chat Cowork
Transport Layer
MCP Servers & Connectors
Open protocol (JSON-RPC 2.0) connecting Claude to live services — databases, APIs, SaaS apps. The "USB-C" of AI connectivity.
Code Chat Cowork
Skills are the universal primitive

A skill is a folder with a SKILL.md file. The exact same skill works in Claude Code, Claude Chat, and Cowork without modification. Build once, use everywhere. Launched Oct 2025 as an open standard.

Plugins are the distribution unit

Plugins bundle multiple skills, connectors, slash commands, and sub-agents into a single installable package. 11+ official plugins (Sales, Finance, Legal, etc.) plus the ability to build your own.

All Extension Points

Seven ways to customize Claude. Click any card for details.

MCP Servers / Connectors

Connect Claude to external tools & APIs. Called "Connectors" in Chat (50+), configured via CLI or .mcp.json in Code. Donated to Linux Foundation Dec 2025.
Code Chat Cowork

Skills

Portable instruction sets (SKILL.md). Claude auto-invokes them based on task context via progressive disclosure. Same open format across all surfaces.
Code Chat Cowork

Plugins

Installable bundles of skills + connectors + agents + hooks + slash commands. 15+ official plugins with 85+ skills. Marketplace distribution.
Code Chat Cowork

Projects / CLAUDE.md

Persistent context. CLAUDE.md files in Code (auto-loaded per repo). Named project spaces with knowledge bases in Chat.
Code Chat Cowork

Subagents / Agent Teams

Specialized AI assistants with isolated context, custom prompts, tool restrictions, and their own models. Run in parallel or in git worktree isolation.
Code Chat Cowork

Hooks

Event-driven automation. Run scripts, call APIs, or invoke Claude at 16+ lifecycle points (PreToolUse, SessionStart, etc.).
Code Chat Cowork

Interactive Apps

Live app interfaces (Slack, Figma, Asana, Canva) embedded directly inside Claude Chat conversations via the MCP Apps extension.
Code Chat Cowork

Which Should I Use?

Common scenarios mapped to the right interface and extension.

"I want Claude to talk to Slack / Asana / our database"
Connect Claude to external services using MCP Servers (in Code) or Connectors (in Chat). Both use the same protocol. 50+ pre-built connectors available in Chat; thousands of community MCP servers for Code.
MCP Server / Connector
"I want Claude to generate Excel reports in a specific format"
Create a Skill with your report template and formatting instructions. Skills work across all three interfaces — the same SKILL.md file will work whether your team uses Chat, Code, or Cowork.
Skill
"I want a reusable /deploy command for my dev team"
Create a Skill at .claude/skills/deploy/SKILL.md and check it into your repo. For distribution across multiple repos, bundle it in a Plugin and host on GitHub.
Skill or Plugin
"Claude should always know our coding conventions"
In Code: put conventions in CLAUDE.md at your repo root (auto-loaded every session). In Chat: create a Project with custom instructions. In Cowork: use folder-specific instructions.
Project / CLAUDE.md
"I want to auto-lint code before Claude commits"
Set up a Hook on PreToolUse or PostToolUse events. Hooks run shell scripts, call APIs, or invoke Claude at 16+ lifecycle points. Claude Code only.
Hook Code only
"Our finance team wants Claude to help with reconciliation"
Install the Finance Plugin in Cowork from the Anthropic marketplace. One-click install, includes specialized skills for journal entries, reconciliation, variance analysis. For Chat users, the same skills are available individually.
Plugin (Cowork) or Skills (Chat)
"I need Claude to process 200 invoices while I'm away"
Claude Cowork is built for autonomous file tasks. Grant it folder access, describe the task, and step away. It uses a coordinator + sub-agent model to parallelize work. You can even schedule recurring tasks.
Cowork
"I want to share a full toolkit across my engineering org"
Build a Plugin that bundles your team's skills, subagents, hooks, and MCP configs. Host on a GitHub marketplace repo. For org-wide enforcement, use managed settings and managed-mcp.json.
Plugin
"Our sales team needs to enrich CRM data from ZoomInfo"
Create a Claude Chat Project with system instructions defining the enrichment workflow (decision tree, conflict rules, logging). Connect HubSpot + ZoomInfo Connectors. Upload reference docs (taxonomy mapping, enum keys). Sales team opens the project and says "enrich contacts missing industry." No code, no terminal.
Project (Chat) + Connectors
"I want Claude to remember our company context across all conversations"
In Code: write a CLAUDE.md at repo root + use auto-memory for cross-session learning. In Chat: create a Project with company docs uploaded as knowledge base. In Cowork: set Global Instructions in Customize menu. Each surface has its own persistence mechanism — use the one matching your interface.
Project / CLAUDE.md / Global Instructions
"Different team members need different Claude workflows"
Build Skills — they're the universal format. The same SKILL.md works in Code (devs), Chat (everyone), and Cowork (ops). Developers build the skill once, then each persona accesses it through their preferred surface. For bundling multiple skills per department, use Plugins.
Skill (build once) Plugin (bundle per team)
"I want to use Figma/Slack inside a Claude conversation"
Interactive Apps in Claude Chat embed live app interfaces directly in your conversation. Built on the MCP Apps extension. Currently available for Slack, Figma, Asana, Canva, Box, Clay, and more.
Interactive Apps (Chat)

Building by Persona

The same outcome requires different tools depending on who's using it. Start with the user, not the technology.

Executives & Biz Dev

Non-technical. Need outcomes, not tools. Won't use a terminal.

Use This Stack
  • Claude Chat Project — persistent context + uploaded reference docs
  • Connectors (MCP) — connect HubSpot, ZoomInfo, Salesforce, etc.
  • Skills — toggle on in Settings → Capabilities for structured workflows
  • Interactive Apps — Slack, Asana, Figma embedded in conversation
Don't Use
  • Claude Code (requires terminal)
  • Hooks, Subagents, Plugins (developer primitives)
  • CLAUDE.md files (requires git repos)
Pattern: Create a Claude Chat Project with system instructions that define the workflow, upload reference docs (taxonomy maps, templates), connect relevant services via Connectors. User just opens the project and chats.
Knowledge Workers

Comfortable with apps but not code. Need autonomous file processing & reports.

Use This Stack
  • Claude Cowork — grant folder access, describe task, step away
  • Plugins — one-click install from marketplace (Finance, Sales, Data, etc.)
  • Connectors — Google Workspace, DocuSign, Snowflake
  • Scheduled Tasks — recurring processing without manual triggers
When to Upgrade to Code
  • Need to modify actual source code
  • Need git integration or CI/CD
  • Batch processing 1000+ files with custom logic
Pattern: Install relevant plugin, point Cowork at a folder, describe the outcome. Cowork's coordinator + sub-agent model handles parallelization automatically.
Developers

Full terminal access. Building the tools, workflows, and integrations others use.

Use This Stack
  • Claude Code — full CLI with filesystem, git, bash, subagents
  • CLAUDE.md — repo conventions auto-loaded every session
  • Custom Skills & Commands — .claude/skills/ and .claude/commands/
  • Hooks — auto-lint, auto-test, compliance logging
  • Agent Teams — parallel agents in independent contexts
  • MCP Servers — connect to databases, APIs, internal tools
Build For Others
  • Build Projects (Chat) for biz dev & execs
  • Build Plugins (Cowork) for knowledge workers
  • Build Skills (universal) for cross-team workflows
Pattern: Developers build the automation and package it for the right surface. A CLI workflow becomes a Chat Project for executives or a Cowork Plugin for ops teams. Skills are the universal format — build once, deploy anywhere.
The Developer-as-Packager Model

The most effective pattern at companies using Claude: developers build the workflows in Code, then package them as Projects (for Chat) or Plugins (for Cowork) so non-technical teammates can use them. The same skill logic works everywhere — the developer's job is choosing the right delivery surface for the audience.

Surface Selection Decision Tree

Evaluate three dimensions — never pick based on persona alone.

Step 1: WHO uses it? (narrows to 2 options, never decides alone)
  Non-technical  → Chat OR Cowork
  Knowledge worker → Chat OR Cowork
  Developer      → Code (or Chat for quick Q&A)

Step 2: WHAT is the task? (picks the surface)
  Conversational, ad-hoc, back-and-forth?
    → Chat Project
    Examples: "look up this contact", quick data questions,
    brainstorming, document review

  Autonomous, batch, describe-outcome-and-walk-away?
    → Cowork Plugin
    Examples: enrich 33K records, process 200 invoices,
    generate quarterly reports, bulk file operations

  Development, code, git, terminal?
    → Code Skill / Command
    Examples: deploy pipeline, run tests, refactor module

Step 3: HOW should it be delivered? (picks the packaging)
  One-off workflow        → Project instructions
  Reusable procedure      → Skill
  Multi-skill bundle      → Plugin

Common Mistake:
  "Non-technical user" ≠ "Chat Project"
  A non-technical user doing batch work needs Cowork.
  A non-technical user doing ad-hoc Q&A needs Chat.
  Persona picks the candidates. Task type picks the winner.

Best Practices

Hard-won patterns from the Claude Code creator and power users. Applicable across all surfaces.

Context Management
  • Keep CLAUDE.md files under 200 lines each for reliable adherence
  • In Code: use /compact before hitting 50% context window
  • Use /clear when switching tasks — don't pollute context
  • Long sessions degrade quality — start fresh rather than push through
  • In Chat: create separate Projects per workflow domain, don't overload one
  • Commit at least once per hour (Code) to create restore points
Prompting Patterns
  • Challenge Claude: "grill me on these changes and don't proceed until I pass your test"
  • Reset mediocre work: "knowing everything you know now, scrap this and implement the elegant solution"
  • Throw compute at it: say "use subagents" to parallelize research
  • Start with planning: always have Claude plan before executing complex work
  • Use a second Claude session to review the first one's plan
  • Write detailed specs to reduce ambiguity — Claude fills gaps with assumptions
Parallelism
  • Code: run 3–5 Claude Code sessions in terminal tabs simultaneously
  • Chat: run 5–10 sessions on claude.ai for research parallelism
  • Agent Teams: multiple independent agents coordinating via shared task list
  • Worktrees: isolated git branches for parallel development without conflicts
  • Multiple uncorrelated context windows > one overloaded context window
  • More tokens = better results. Separate contexts make it even better.
Skills Architecture
  • If you do it more than once a day, make it a skill
  • Skills are the universal primitive — same SKILL.md works in Code, Chat, and Cowork
  • Skills auto-invoke based on task context (progressive disclosure)
  • Full content only loads when relevant — descriptions are always in context
  • Character budget: 15,000 chars default. Override with SLASH_COMMAND_TOOL_CHAR_BUDGET
  • Resolution order: Skill (inline) → Agent (separate context) → Command (explicit / only)
Verification
  • Most important tip: always give Claude a way to verify its own work
  • In Code: use test suites, lint, type-check after every significant change
  • Use a separate context window to review work (one agent builds, another reviews)
  • In Chat: ask Claude to "prove to me this works" or "what could go wrong?"
  • In Cowork: review coordinator's draft before approving — it shows its work
  • Don't trust long sessions — quality degrades. Fresh session = fresh eyes.
MCP Best Practices
  • Let Claude use all your tools via MCP — Slack, BigQuery, Sentry, CRM, etc.
  • For contacts already enriched via a linked service, read from the link first — saves API quota
  • In Chat: use Connectors (one-click). In Code: use .mcp.json
  • ZoomInfo, HubSpot, Salesforce, and 50+ services have pre-built connectors for Chat
  • Three scopes: Project (.mcp.json), User (~/.claude.json), Agent (frontmatter)
  • Use Tool Search (ENABLE_TOOL_SEARCH=auto:N) when MCP tools exceed 10% context
Daily Habits from the Claude Code Creator (Boris Cherny)

Update Claude Code daily. Share a single CLAUDE.md with your team and check it into git. Tag @claude on PRs to update CLAUDE.md. Pre-approve permissions instead of using --dangerously-skip-permissions. Use Opus with thinking for everything important. Use Haiku for fast codebase searches.

Technical Reference

Frontmatter specs, memory systems, and configuration hierarchy for builders.

Agent vs Command vs Skill
Dimension         Agent           Command          Skill
───────────────────────────────────────────────────────────
Context           Separate        Inline (main)    Inline*
User-invocable    No /menu        Yes /name        Yes /name
Auto-invoked      Yes (desc)      Never            Yes (desc)
Own context       Yes (isolated)  No               No*
Model override    Yes             Yes              Yes
Tool restrict     Yes             Yes              Yes
Hooks             Yes             No               Yes
Memory            Yes             No               No
Preload skills    Yes             No               No
MCP servers       Yes             No               No

When to use:
Agent  → autonomous multi-step, context isolation
Command → user-initiated entry point, orchestration
Skill  → reusable knowledge, auto-invoke on intent
Memory System (4 layers)
System        Who Writes   Who Reads      Scope
──────────────────────────────────────────────────
CLAUDE.md     You          Claude + all   Project
                           agents
Auto-memory   Claude       Claude only    Per-project
                           (auto)                   per-user
/memory cmd   You          Claude only    Per-project
                                          per-user
Agent memory  The agent    That agent     Configurable
              itself       only

CLAUDE.md Loading (monorepos):
UP   → walks from cwd to root, loads all
DOWN → subdirectory CLAUDE.md loads lazily
         (only when Claude reads files there)
SIBLING → never loads

Agent Memory Scopes:
user    → ~/.claude/agent-memory/<name>/
project → .claude/agent-memory/<name>/
local   → .claude/agent-memory-local/<name>/
Skill Frontmatter (10 fields)
---
name: quarterly-report        # display name + /slash
description: Generate reports  # auto-discovery trigger
argument-hint: <quarter>      # autocomplete hint
allowed-tools: Read, Write     # no permission prompts
model: opus                    # sonnet | opus | haiku
context: fork                  # isolated subagent
agent: general-purpose         # subagent type (w/ fork)
user-invocable: true           # show in / menu
disable-model-invocation: false
hooks:                         # lifecycle hooks
  PostToolUse:
    - matcher: "Edit"
      command: "npm run lint"
---
