Recipe: Build a Custom Agent#

What You Will Build#

A code review agent that team members invoke to get thorough, structured PR reviews. When someone runs claude --agent code-reviewer "Review PR #42", the agent checks out the branch, reviews every changed file, and produces a written summary covering bugs, security issues, and style violations.

Agents are markdown files that define a specialized persona and workflow for Claude. Unlike skills (which are short instruction sets), agents provide a complete system prompt that shapes how Claude approaches an entire task.

Prerequisites#

  • Claude Code installed
  • A Git repository with pull requests to review
  • GitHub CLI (gh) installed (for PR checkout) or manual branch switching
  • Basic familiarity with Agents concepts

Step 1: Create the Agent File#

Agent files live in .claude/agents/ in your project root. Each file defines one agent.

bash 
<span class="hljs-built_in">mkdir</span> -p .claude/agents
<span class="hljs-built_in">touch</span> .claude/agents/code-reviewer.md

Your project structure:

plaintext 
my-app/
  .claude/
    agents/
      code-reviewer.md    &lt;-- your new agent
  src/
  package.json

Step 2: Write the Agent Prompt#

Open .claude/agents/code-reviewer.md and write the full agent definition. The file content becomes Claude's system prompt when the agent is invoked.

markdown 
---
name: code-reviewer
description: Performs thorough code reviews on pull requests
allowed-tools:
<span class="hljs-bullet">  -</span> Bash
<span class="hljs-bullet">  -</span> Read
<span class="hljs-bullet">  -</span> Grep
<span class="hljs-section">  - Glob
---</span>

<span class="hljs-section"># Code Review Agent</span>

You are a senior code reviewer. When given a PR number or branch name,
perform a thorough review following this process.

<span class="hljs-section">## Review Process</span>

<span class="hljs-section">### 1. Gather Context</span>

<span class="hljs-bullet">-</span> Check out the PR branch: <span class="hljs-code">`gh pr checkout &lt;number&gt;`</span>
<span class="hljs-bullet">-</span> List all changed files: <span class="hljs-code">`gh pr diff --name-only`</span>
<span class="hljs-bullet">-</span> Read the PR description: <span class="hljs-code">`gh pr view &lt;number&gt;`</span>

<span class="hljs-section">### 2. Review Each File</span>

For every changed file, check for:

<span class="hljs-strong">**Bugs**</span>
<span class="hljs-bullet">-</span> Null/undefined access without guards
<span class="hljs-bullet">-</span> Off-by-one errors in loops or slices
<span class="hljs-bullet">-</span> Race conditions in async code
<span class="hljs-bullet">-</span> Resource leaks (unclosed connections, file handles)

<span class="hljs-strong">**Security**</span>
<span class="hljs-bullet">-</span> User input used without sanitization
<span class="hljs-bullet">-</span> SQL injection vectors
<span class="hljs-bullet">-</span> Secrets or credentials in code
<span class="hljs-bullet">-</span> Missing authentication or authorization checks

<span class="hljs-strong">**Style**</span>
<span class="hljs-bullet">-</span> Functions longer than 50 lines
<span class="hljs-bullet">-</span> Deeply nested conditionals (3+ levels)
<span class="hljs-bullet">-</span> Magic numbers without named constants
<span class="hljs-bullet">-</span> Missing or misleading comments

<span class="hljs-section">### 3. Write the Review</span>

Produce a structured review with these sections:

<span class="hljs-section">#### Summary</span>
One paragraph describing the overall change and its quality.

<span class="hljs-section">#### Issues Found</span>
A numbered list, each with:
<span class="hljs-bullet">-</span> <span class="hljs-strong">**Severity**</span>: Critical / Warning / Suggestion
<span class="hljs-bullet">-</span> <span class="hljs-strong">**File**</span>: path and line number
<span class="hljs-bullet">-</span> <span class="hljs-strong">**Description**</span>: what the issue is
<span class="hljs-bullet">-</span> <span class="hljs-strong">**Fix**</span>: how to resolve it

<span class="hljs-section">#### Positive Notes</span>
Call out well-written code, good patterns, and thoughtful design.

<span class="hljs-section">#### Verdict</span>
One of: APPROVE, REQUEST<span class="hljs-emphasis">_CHANGES, or NEEDS_</span>DISCUSSION.

Step 3: Understand the Agent Format#

The agent file has two parts:

  • Frontmatter (between --- markers) -- Metadata that tells Claude which tools the agent can use and how to identify it.

    • name -- The identifier you use on the command line.
    • description -- Shown when listing agents.
    • allowed-tools -- Restricts tools to only what the agent needs. The code reviewer needs Bash for git commands, Read to view files, and Grep/Glob to search the codebase.

  • Body (the markdown below the frontmatter) -- The system prompt Claude follows. Write this as direct instructions: what to do, in what order, and how to format the output.

Step 4: Invoke the Agent#

Run the agent from your project directory:

bash 
claude --agent code-reviewer <span class="hljs-string">&quot;Review PR #42&quot;</span>

Claude will:

  1. Load the code-reviewer.md agent prompt.
  2. Use gh pr checkout 42 to switch to the PR branch.
  3. Read every changed file.
  4. Produce a structured review following your template.

You can also pass context:

bash 
claude --agent code-reviewer <span class="hljs-string">&quot;Review PR #42, focus on the auth changes&quot;</span>

Step 5: Customize for Your Team#

Add team-specific rules to the agent body. For example:

markdown 
<span class="hljs-section">## Team Standards</span>

<span class="hljs-bullet">-</span> All API endpoints must have JSDoc comments.
<span class="hljs-bullet">-</span> Database queries must use parameterized statements (no string interpolation).
<span class="hljs-bullet">-</span> React components must have PropTypes or TypeScript interfaces.
<span class="hljs-bullet">-</span> Test coverage must not decrease (check with <span class="hljs-code">`npm run coverage`</span>).
<span class="hljs-bullet">-</span> No <span class="hljs-code">`console.log`</span> in production code (use the logger utility).

<span class="hljs-section">## Forbidden Patterns</span>

Flag these immediately as Critical issues:
<span class="hljs-bullet">-</span> <span class="hljs-code">`eval()`</span> or <span class="hljs-code">`new Function()`</span>
<span class="hljs-bullet">-</span> <span class="hljs-code">`dangerouslySetInnerHTML`</span> without sanitization
<span class="hljs-bullet">-</span> <span class="hljs-code">`// TODO`</span> or <span class="hljs-code">`// FIXME`</span> in new code (must be resolved before merge)
<span class="hljs-bullet">-</span> Direct database access outside the repository layer

These rules make the agent enforce your specific coding standards, not just generic best practices.

Verify It Works#

Your agent is working correctly when:

  • Running claude --agent code-reviewer "Review PR #42" produces a structured review.
  • The review covers all changed files, not just the first one.
  • Issues include file paths and line numbers.
  • The verdict section gives a clear APPROVE, REQUEST_CHANGES, or NEEDS_DISCUSSION.
  • Team-specific rules are enforced (if you added them).
